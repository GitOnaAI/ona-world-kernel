# Onboarding: AI Game Board project

This skill extracts context from THIS repo and populates the Ona Knowledge Base so the
Documentation Writer and the Discovery/GDD pipeline have rich context about what the game
already contains. In this template, **the game repo is the current working directory**:
the session runs inside the clone (kernel or fork), so there is no separate repo to locate.

## Step 1: receive arguments

The onboarding command generated by the Ona UI banner includes the arguments:

```
/onboarding-game
project_id: <UUID>
api_key: ona_sk_<token>   (included automatically by the banner)
```

Extract from the ARGUMENTS:
- `project_id`: required. If missing, ask the user for it (it is shown in the Ona project
  onboarding banner).
- `api_key`: optional; if present, use it for ALL MCP/HTTP calls instead of environment
  variables. If absent, try `ONA_PROD_API_KEY`, then `ONA_API_KEY`, then ask the user.

## Step 1b: detect the environment and validate access

Try project access in this order and stop at the first 200:

```bash
# 1. Prod
curl -s "https://onaai.cloud/api/v1/projects/{project_id}" \
  -H "Authorization: Bearer {api_key}" | grep -q '"id"' && echo "prod"

# 2. Staging
curl -s "https://staging.onaai.cloud/api/v1/projects/{project_id}" \
  -H "Authorization: Bearer {api_key}" | grep -q '"id"' && echo "staging"

# 3. Local
curl -s "http://localhost:8000/api/v1/projects/{project_id}" \
  -H "Authorization: Bearer {api_key}" | grep -q '"id"' && echo "local"
```

Record `BASE_URL` and `API_KEY` for all following steps. If no environment responds,
inform the user and stop.

Optionally confirm the project type:
```bash
curl -s "${BASE_URL}/api/v1/projects/{project_id}" -H "Authorization: Bearer ${API_KEY}" \
  | grep -q '"template_type":"ai_game_board"' && echo "ai_game_board confirmed"
```
If `template_type` is not `ai_game_board`, warn the user (it can be intentional, a game
project on a generic template, but confirm before proceeding).

## Step 1c: ensure the development environment is configured

If the `ona_*` MCP tools are not already available in this session, set up the connection
as documented in "Connecting to Ona" in the root `CLAUDE.md`: copy `.mcp.json.example` to
`.mcp.json` and fill in the API key (already resolved above), and confirm `.mcp.json` is
listed in `.gitignore`. Alternatively, Ona's setup-script does the same thing end to end:
`GET ${BASE_URL}/api/v1/projects/{project_id}/setup-script`, executed inside this repo.
A restart of the session may be needed for the MCP tools to appear; until then, every
`ona_*` call below also works via the HTTP fallback (JSON-RPC POST to
`${BASE_URL}/api/v1/mcp-server/mcp` with the same tool name and arguments, Authorization:
Bearer ${API_KEY}).

## Step 2: validate this repo

`GAME_REPO_PATH` is the repo root (the current working directory).

```bash
test -d "src/sim" && test -f "src/sim/data.ts" \
  && echo "valid repo: Ona World Kernel (or compatible fork)" \
  || echo "WARNING: src/sim/data.ts not found; diverged fork, map best-effort"
```

If the structure does not match the kernel (a diverged fork), do not block: proceed in
best-effort mode (read what exists, catalog what is found, and mark missing categories as
"not found in this repo" instead of inventing paths).

## Step 3: map the repo documentation

Read the context files, in priority order:

1. **CLAUDE.md, root AND per directory.** The kernel has one `CLAUDE.md` per subarea, each
   documenting that module's conventions (e.g. `src/sim/content/CLAUDE.md` explains how
   content is organized per zone):
   ```bash
   find . -name "CLAUDE.md" \
     -not -path "*/node_modules/*" -not -path "*/dist*/*" -not -path "*/.git/*"
   ```
2. **README and product/design docs:**
   ```bash
   ls README.md AGENTS.md 2>/dev/null
   find docs -maxdepth 2 -iname "*.md" 2>/dev/null
   ```
   Pay special attention to `docs/design/*.md` when present: that is where closed design
   decisions live, and reading them prevents a later GDD from reopening them.
3. **Technical configuration summary:** engine/rendering dependencies in `package.json`
   (useful context for performance RNFs in a GDD).

## Step 4: catalog the content library

**This is the core of the game onboarding**: without it, every GDD generated later
reinvents content that already exists. For each category below, **confirm the real path in
this repo before writing the note** (forks diverge from the reference kernel); the paths
given are the kernel defaults, a verified starting point, not a guarantee.

Each category becomes **one separate note** via `ona_add_note` (or the HTTP fallback from
Step 1c). Each note documents: **what exists, where it lives, how to reuse it** (an index
for the GDD agent, not the full content).

### 4a. Zones
Where: `src/sim/content/zone*.ts` (one per zone; confirm how many exist). Each zone
defines its local mobs/NPCs/quests/loot, merged into the central registry
(`src/sim/data.ts`). `title="Content Library: Zones"`

### 4b. Rigged creature families
Mechanics (mob templates, `family`) in `src/sim/data.ts` (`MOBS`) and per zone; the visual
rig (family to glTF asset + animation clips) in `src/render/characters/manifest.ts`
(`FAMILY_KEYS`/`VisualDef`/`ClipMap`). Reusing an existing family means changing
stats/loot in the sim registry without touching render; a new family also requires a rig
in `manifest.ts`. `title="Content Library: Creature Families"`

### 4c. Items, item sets, and recipes
Base items in `src/sim/content/items.ts` (`BASE_ITEMS`); equipment sets in `item_sets.ts`;
profession recipes in `recipes.ts` + `professions.ts`; delve/dungeon-specific items may
live in `src/sim/content/delves/items.ts`. `title="Content Library: Items, Sets, Recipes"`

### 4d. Classes and talents
`src/sim/content/classes.ts` (abilities per class), `talents.ts` plus per-class variants.
`title="Content Library: Classes and Talents"`

### 4e. Dungeons and delves
Fixed dungeons: `src/sim/content/dungeons.ts`. Delves (procedural/modular instance
content): `src/sim/content/delves/` (subdivided by system: `mobs.ts`, `items.ts`,
`shop.ts`, `companions.ts`, `affixes.ts`, ...). `title="Content Library: Dungeons and Delves"`

### 4f. NPCs, vendors, and quests
Central registry in `src/sim/data.ts` (`NPCS`, `QUESTS`); vendor stock in
`src/sim/vendor_stack.ts`; delve-specific vendors in `src/sim/content/delves/shop.ts`.
`title="Content Library: NPCs, Vendors, Quests"`

### 4g. Procedural generators
One block per generator, each with where it lives and what it produces:
- **Villages/settlements**: `src/render/props.ts` (the `village` kit composed from
  reusable glTF pieces).
- **Terrain**: `src/render/terrain.ts` + `terrain_region_core.ts` + `impact_terrain.ts`.
- **Vegetation**: `src/render/foliage.ts` (plus the offline pipeline
  `scripts/assets/build_foliage.mjs`).
- **Weather**: `src/render/weather.ts`.
- **Icons**: `src/ui/icons.ts` (ability/item/aura icons painted procedurally on canvas;
  ids without a manual recipe fall back automatically, so new content never lacks an icon).
- **Audio**: `scripts/gen_npc_lines.mjs` (NPC lines via TTS from `src/sim/data.ts`),
  `scripts/gen_npc_voices.mjs`, `scripts/gen_sfx.mjs`, `scripts/render_music.mjs` +
  `src/game/music.ts`. All offline, not part of the standard build.
`title="Content Library: Procedural Generators"`

## Step 5: publish the summary document

```
ona_push_doc(
  title="Game Context: <name>",
  content="<500-800 word summary covering game vision, genre, engine/stack, and a quick
    index of the categories cataloged in Step 4 (zones, creatures, items, classes,
    dungeons/delves, NPCs/quests, procedural generators) with each one's root path>",
  source_path="onboarding/<project_id>/game-context-summary",
  project_id="<project_id>"
)
```

Same HTTP fallback pattern as Step 4 (tool name: `ona_push_doc`).

## Step 6: finish the onboarding

```
ona_complete_onboarding(project_id="<project_id>")
```

Same HTTP fallback pattern (tool name: `ona_complete_onboarding`).

After a return with `{"project_id": "...", "onboarding_status": "done"}`:
- Tell the user the onboarding is complete.
- The banner in the Ona UI disappears automatically (5s polling).
- The project board is ready: the Discovery/GDD agent now has context about what already
  exists in the game and will prioritize reuse over new content.

## Common errors

- **`Project not found` in every environment**: the `api_key` belongs to another tenant.
  Ask the user to generate a new key in Settings, API keys, in the correct org.
- **`400 Project does not require onboarding`**: the project was created without an
  onboarding sub_type, or is not `ai_game_board`.
- **MCP tools do not appear**: use the HTTP fallback from Step 1c; it is equivalent.
- **No `src/sim/data.ts` in this repo**: a diverged fork or a from-scratch game. Do not
  block: map best-effort (Step 2) and mark missing categories explicitly in the Step 4
  notes instead of inventing paths.
